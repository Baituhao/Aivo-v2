# 关键技术决策记录（ADR）

记录 Aivo-v2 课设过程中的重要技术选型与思考，便于答辩时说明「为什么这样选」。

---

## ADR-001：LLM 大脑用 API，而非自托管开源模型（如 Hermes）

### 背景
数字人需要一个「大脑」做对话、意图识别、工具调用、工作流规划。
候选：① 自己在 H20 上跑开源模型（Hermes 3 / Qwen 本地版）；② 调用云端 API（Claude / Qwen / DeepSeek）。

### 对比

| 维度 | 自托管开源（Hermes 等） | API（Claude 等） |
|------|------------------------|------------------|
| 部署 | 自己部署维护 | 直接调用，无需部署 |
| 费用 | 免费（已付算力） | 按 token 付费 |
| 隐私 | 数据不出本地 | 数据发往服务商 |
| 断网可用 | 可以 | 不行 |
| 能力上限 | 中等（7B-70B） | 顶级推理 |
| 工具调用稳定性 | 一般，需调教 prompt | 原生强，JSON 稳定 |
| 显存占用 | 吃 H20 一大块 | 0 显存 |

### 决策
**LLM 用 API。** 把宝贵的 H20 显存全留给「没有 API、只能自己跑」的算法
（数字人渲染、自定义音色 TTS）。开源小模型在工具调用和结构化 JSON 输出上
最容易翻车，而这恰恰是本项目的核心。

### 何时该重新考虑自托管
- 课设明确要求「全本地化 / 数据不出域」
- 想在答辩展示「自部署 + 微调开源大模型」这个亮点
- 任务简单到 7B 模型够用，且不想付 API 费

## ADR-002：开发模式 — 直接用 Claude（单智能体）vs Workflow 编排（多智能体）

> 注：这是「开发本项目时如何使用 AI 助手」的工作方式选择，
> 与 ADR-001（项目运行时的 LLM 大脑）是两件不同的事。

### 两种模式的本质

- **直接用 Claude**：一个大脑串行干活，上下文全程共享、连贯记忆。
- **Workflow 编排**：一个指挥官派生几十个独立子 agent 并行干活，最后汇总。

### 对比

| 维度 | 直接用 Claude | Workflow 编排 |
|------|--------------|---------------|
| 并行度 | 串行，一次一件 | 约 10-16 个 agent 并行 |
| 上下文 | 全部共享 | 各子 agent 独立隔离 |
| Token 消耗 | 低 | 高（可能 10 倍以上） |
| 适合场景 | 对话、单点修改、连贯任务 | 大规模扫描、多视角审查、批量迁移 |
| 控制流 | 随机应变 | 脚本写死（循环/分支/扇出） |

### 决策
**搭脚手架、调试单个服务 → 直接用 Claude**（连贯、省钱）。
将来遇到以下场景再开 Workflow：
- 整个 DSW 后端代码全面安全审计
- 对比 SadTalker / Wav2Lip / MuseTalk 三方案，各派 agent 深挖打分
- 批量给所有 API 端点加鉴权和限流
